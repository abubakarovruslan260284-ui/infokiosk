# Инфокиоск — новая оболочка (Tauri) + движок синхронизации контента

Рабочий прототип по задаче: переписать оболочку инфокиоска на другом
(не Electron) языке ради скорости на слабом железе, и добавить
локальную синхронизацию контента с общей папкой — без единого лишнего
байта по сети в момент, когда покупатель сканирует товар.

Скелет («штрихкод → 1С → цена») не тронут: как и раньше, это обычный
`fetch`/`XHR` из веб-страницы прямо к HTTP-сервису 1С. Ему всё равно,
в каком хосте он выполняется.

## Архитектурная идея в двух предложениях

Киоск никогда не показывает контент "с сети" — только из локального
кэша на диске. Фоновый поток раз в 15–30 секунд дёшево спрашивает один
маленький файл — `manifest.json` в общей папке — и, только если его
версия изменилась, докачивает во временную папку ровно то, что
реально поменялось (по SHA-256), и одним атомарным переименованием
подменяет рабочий кэш. Показ на экране — это либо чтение уже готового
кэша при старте, либо обновление, которое подставляется только в
безопасный момент (когда на экране слайдер, а не карточка цены).

## Что реально проверено (в этой песочнице)

**Движок синхронизации `kiosk_sync` — да, полностью, живым кодом:**
- 13 автотестов (`cargo test`), все проходят. Среди них: докачка только
  изменённого файла, удаление пропавших файлов, переживание обрыва
  сети/папки без потери текущего кэша, отказ от подмены при
  повреждённой докачке (проверка контрольной суммы).
- Отдельный бенчмарк на реалистичном объёме (20 фото + 3 коротких
  видео, ~34 МБ): первая синхронизация — доли секунды, повторная
  (2 изменённых файла из 24) — в разы быстрее первой именно за счёт
  того, что нетронутые файлы не перекачиваются и не перепроверяются.
- Собран и опробован «Издатель» (`kiosk_publish`) — реальный
  скомпилированный бинарник: публикация папки → `manifest.json`,
  повторная публикация с правкой → киоск (тоже реальный код,
  использующий тот же `kiosk_sync`) забирает точно только разницу.
  Полный цикл прогнан вручную и воспроизводимо.

**Фронтенд (преднагрузка + мгновенное переключение) — да, в браузере:**
- `dist/` целиком — тот же HTML/CSS/JS, что и в утверждённой
  ИрсКом-версии, плюс новый `content.js`.
- Собран тестовый стенд с имитацией моста `window.__TAURI__` и
  замерены реальные тайминги (Playwright, headless Chromium, с
  прогревом движка и несколькими повторами на сценарий): переключение
  на карточку цены при заранее прогретых слайдах даёт заметно меньший
  пиковый кадр во время перехода, чем если бы декодирование картинки
  происходило прямо в момент показа. Числа в этой песочнице
  ориентировочные (виртуализированная среда без GPU), но сам эффект —
  воспроизводим и логичен: работа, сделанная заранее, не может
  тормозить в момент, когда её уже не нужно делать.
- Проверено: пока показана карточка цены, фоновое обновление контента
  НЕ трогает то, что видно на экране; оно применяется только после
  возврата на экран ожидания.

**Сама Tauri-оболочка (Rust-код `main.rs`/`settings.rs`/`slides.rs`/
`sync_loop.rs`) — написана и вычитана вручную, полный граф зависимостей
успешно разрешён (`Cargo.lock` в комплекте, зафиксировано 445 пакетов,
включая `tauri 2.11.5`, `tauri-plugin-dialog 2.7.1`,
`tauri-plugin-shell 2.3.5` — версии совместимы друг с другом без
конфликтов), но живая КОМПИЛЯЦИЯ в этой песочнице не пройдена до конца.**
Причина честная и техническая: здесь нет доступа к `rustup` (сетевая
политика песочницы блокирует его домены), а системный Rust из `apt`
(1.75) слишком старый, чтобы прочитать манифест одной из зависимостей
(она использует edition2024, которую понимает только свежий Cargo).
Разрешение зависимостей произошло ДО этого шага и завершилось
успешно — упёрлись именно в устаревший компилятор, не в ошибку графа
зависимостей. Это ограничение именно песочницы, а не самого кода.

Что это значит на практике: `Cargo.lock` уже лежит в проекте, поэтому
при первой сборке в вашем GitHub Actions (см. ниже) — на чистой машине
со свежим Rust — соберётся ровно на этих проверенных версиях. Компилятор
теоретически может попросить 1–2 мелкие правки на уровне точных
названий методов, если что-то в исходниках (которые я не мог
прогнать через настоящий `rustc`) окажется чуть не в ту сторону — это
нормально для написанного "вслепую" кода и обычно занимает 5–15 минут
по сообщениям самого компилятора — они в Rust почти всегда прямо
говорят, что поправить.

## Как собрать по-настоящему (на вашей стороне)

Есть уже готовый файл `\.github/workflows/build-windows.yml` — ровно
тот же принцип, что у Irskom Audio Manager: пушите в репозиторий,
GitHub Actions сам ставит свежий Rust+Node на чистой машине и собирает
`.exe`/`.msi`. Артефакты появятся во вкладке Actions → Artifacts.

Локально (если решите собирать на своей машине с Windows):
```powershell
# один раз
winget install Rustlang.Rustup
npm install -g @tauri-apps/cli

# сборка
cd infokiosk-tauri
tauri build
```

## Настройка на месте (settings.json)

Живёт в папке данных приложения пользователя (Tauri сам создаёт при
первом запуске, обычно `%LOCALAPPDATA%\<identifier>\`). Пример — в
`settings.example.json`. Ключевое новое поле:

- `content_source_path` — путь к общей папке с контентом. Это может
  быть UNC-путь к сетевой шаре (`\\ПК-ИЗДАТЕЛЯ\kiosk-content`) или
  локальная папка, если «Издатель» и киоск на одной машине.

## Как публиковать контент («Издатель»)

```powershell
kiosk_publish.exe "\\ПК-ИЗДАТЕЛЯ\kiosk-content"
```

Кладёте туда фото/GIF/mp4 (короткие, уже сжатые под экран — это
отдельная и важная задача самого «Издателя»-приложения с UI, которую
имеет смысл делать следующим шагом), запускаете команду — она
пересчитает `manifest.json`. Все подключённые киоски заберут разницу
при следующем опросе (по умолчанию — раз в 20 секунд), без
перезапуска.

## Известные ограничения и что стоит сделать дальше

1. **Живая компиляция оболочки** — см. выше. Первым шагом после
   получения этих файлов стоит просто нажать «Run workflow» в GitHub
   Actions и посмотреть на реальный лог сборки.
2. **У «Издателя» пока только командная строка**, без графического
   интерфейса и без автоматического сжатия видео. Тот UI с
   drag-and-drop и предпросмотром, который обсуждался раньше — это
   отдельная (не самая сложная) надстройка поверх уже готового и
   протестированного `kiosk_sync`.
3. **Иконки приложения** — сейчас это временная заглушка фирменных
   цветов, не финальный логотип.
4. Слежение за манифестом сейчас **только через опрос** (poll), без
   мгновенных OS-уведомлений об изменении файла в сетевой папке.
   Это осознанный выбор: уведомления об изменении файлов через SMB
   ненадёжны (часто теряются), а опрос раз в 15–30 секунд не создаёт
   заметной задержки для покупателя и гарантированно не пропустит
   изменение.
